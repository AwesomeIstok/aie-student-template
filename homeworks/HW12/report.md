# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Датасет `S12-hw-dataset.csv` — временной ряд с почасовыми наблюдениями. Прогнозировалась целевая переменная target.
- Разбиение `train / validation / test` по времени `70 / 15 / 15`.
- Сравнивались подходы `B1 (Naive)`, `B2 (Moving Average)`, `B3 (Ridge)`, `R1 (GRU)`.
- Лучший подход `R1 (GRU)` с валидационным `MAE = 5.72`, превзошёл все baseline-модели.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: 3024 строк  / 648 строк / 648 строк
- Какие доли или интервалы использовались: 70 (train) / 15 (validation) / 15 (test)
- Почему `random split` здесь некорректен: нельзя перемешивать временные ряды, потому что они меют временную зависимость. Нельзя по будущему предсказывать прошлое.

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: lag_1, lag_7, lag_14, lag_28
- Какие rolling-признаки использовались: rolling_mean_7, rolling_std_7
- Какие календарные признаки использовались: dayofweek, month
- Как обрабатывались пропуски после построения лагов/rolling: с помощью dropna()
- Использовалось ли масштабирование и где именно оно обучалось: StandartScaler обучен только на train, применен к val/test. Для GRU - отдельный скейлер для target

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`): последнее известное значение (lag_1)
- B2 (`moving-average`): скользящее среднее за 7 шагов (rolling_mean_7)
- B3 (`ridge-lag-features`): Ridge регрессия на лаговых, rolling и календарных признаках
- R1 (`gru-forecast`): GRU на оконном представлении ряда window_size=28

Дополнительно:

- Основная метрика выбора лучшей модели: MAE на validation
- Дополнительные метрики: RMSE, MAPE
- `window_size` для `GRU`: 28
- Batch size: 64
- Максимальное число эпох: 30
- Optimizer: Adam
- Learning rate: 0.0005
- Как сохранялась лучшая модель: сохранение с помощью state_dict

## 7. Анализ

(8-15 предложений)

Нужно прокомментировать:

- Наивный baseline (B1) оказался сильным — это характерно для временных рядов с высокой автокорреляцией, где следующее значение сильно зависит от предыдущего.
- Moving Average (B2) показал худший результат, скорее всего из-за излишнего сглаживания и потери чувствительности к резким изменениям.
- Ridge на лаговых признаках (B3) показал результат, близкий к Naive, но не превзошёл его. Это может означать, что линейная комбинация лагов не добавляет существенной информации по сравнению с простым lag_1.
- GRU (R1) показала лучший результат благодаря способности улавливать нелинейные зависимости и долгосрочные паттерны в последовательности. window_size (28 шагов) позволил модели учесть сезонность.
- Тренд и сезонность заметны в данных. GRU успешно их учитывает, в то время как простые модели реагируют только на локальные изменения.
- Утечки могли бы быть в некорректных скользящих окнах center=True, нормализации до split, случайном перемешивании
- Выбор по validation, а не по test, таков, что test должен использоваться только один раз для финальной оценки, иначе происходит подгонка под тест и переобучение.
- Наиболее показательные ошибки модели наблюдаются в точках резких скачков значения. Здесь все модели, допускают наибольшие отклонения.

## 8. Итоговый вывод

(3-7 предложений)

- Как базовый я взял бы Naive baseline (B1). Он прост, интерпретируем и часто даёт сильный результат.
- Достаточно простого baseline на признаках, если ряд имеет высокую автокорреляцию, мало данных или требования к скорости прогноза критичны.
- Смысл переходить к `GRU`/другим нейросетям есть при наличии сложных нелинейных паттернов, достаточном объёме данных и требованиях к максимальной точности.
- Всегда нужно использовать temporal split, масштабируйте только на train, и никогда не выбирать модель по test, потому что это нарушает принцип независимой финальной оценки.
